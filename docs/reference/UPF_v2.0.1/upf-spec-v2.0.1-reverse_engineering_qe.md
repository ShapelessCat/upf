# UPF v2.0.1 QE Practical Authoritative Reference

## Scope

This document defines the UPF v2.0.1 behavior reverse-engineered from Quantum
ESPRESSO's practical reader and writer in `upflib/read_upf_new.f90` and
`upflib/write_upf_new.f90`.

This is not a restatement of the prose specification alone. When the prose,
example files, and QE implementation differ, this document follows the QE
implementation and the real UPF v2.0.1 files inspected in:

- `examples/SSSP_1.3.0_PBE_efficiency/UPF_2.x/`
- `/Users/shapeless_cat/MzProjects/pp_input_projects/q-e/pseudo/`

## Document Form

An UPF v2.0.1 document is rooted at `UPF`, normally with
`version="2.0.1"`. Section names are uppercase. Header data is encoded primarily
as attributes on `PP_HEADER`. Repeated families usually use numbered element
names such as `PP_BETA.1` and `PP_CHI.1`.

`PP_INFO` is written for human readers but is outside the structured reader
contract of `read_upf_new.f90`.

Example document skeleton:

```xml
<UPF version="2.0.1">
  <PP_INFO>...</PP_INFO>
  <PP_HEADER element="Si" pseudo_type="NC" mesh_size="4" number_of_proj="1" />
  <PP_MESH>...</PP_MESH>
  <PP_LOCAL>...</PP_LOCAL>
  <PP_NONLOCAL>...</PP_NONLOCAL>
  <PP_PSWFC>...</PP_PSWFC>
  <PP_RHOATOM>...</PP_RHOATOM>
</UPF>
```

## Practical Section Order

QE's practical read path is driven by this order:

1. `UPF`
2. `PP_HEADER`
3. `PP_MESH`
4. `PP_NLCC` when `core_correction` is true
5. `PP_LOCAL` when `is_coulomb` is false
6. `PP_SEMILOCAL` when `pseudo_type == "SL"`
7. `PP_NONLOCAL` unless `number_of_proj = 0`
8. `PP_PSWFC`
9. `PP_FULL_WFC` when `has_wfc` is true
10. `PP_RHOATOM`
11. `PP_TAUMOD` and `PP_TAUATOM` when `with_metagga_info` is true
12. `PP_SPIN_ORB` when `has_so` is true
13. `PP_PAW` when `is_paw` is true
14. `PP_GIPAW` when `has_gipaw` is true

The header must be processed before mesh-sized or count-sized arrays because
later allocations depend on `mesh_size`, `number_of_wfc`, and `number_of_proj`.

## Field Specifications

### PP_INFO

`PP_INFO` is descriptive metadata. QE writes it, but the practical reader does
not interpret it into the in-memory UPF object. It may contain free text and an
optional nested `PP_INPUTFILE`.

Example:

```xml
<PP_INFO>
  Generated by atomic test generator
  Author: Example Author
  Pseudopotential type: NC
  Element: Si
  Functional: PBE
  <PP_INPUTFILE>
    title = 'Si example'
    pseudo_type = 'NC'
  </PP_INPUTFILE>
</PP_INFO>
```

### PP_HEADER

`PP_HEADER` carries machine-readable dataset metadata as attributes. The
practical header surface includes:

- identity: `generated`, `author`, `date`, `comment`, `element`
- classification: `pseudo_type`, `relativistic`
- flags: `is_ultrasoft`, `is_paw`, `is_coulomb`, `has_so`, `has_wfc`,
  `has_gipaw`, `paw_as_gipaw`, `core_correction`, `with_metagga_info`
- scalars and counts: `functional`, `z_valence`, `total_psenergy`,
  `wfc_cutoff`, `rho_cutoff`, `l_max`, `l_max_rho`, `l_local`, `mesh_size`,
  `number_of_wfc`, `number_of_proj`

Real files use both `T` / `F` and `true` / `false` spellings for logical
attributes. `l_local` can be negative. The `relativistic` attribute uses
`no`, `scalar`, or `full` in QE-written files (matching the `CHARACTER(len=6)`
field in `pseudo_types.f90`).

QE's writer always emits all header attributes. QE's reader silently defaults
any missing attribute (see "Attribute Default Behavior" below). The identity
attributes (`generated`, `author`, `date`, `comment`) are not used by QE's
computational code and exist only for provenance tracking. In particular, a
missing `comment` attribute is read as the empty string.

Example:

```xml
<PP_HEADER
  generated="Example generator"
  author="Example Author"
  date="2026-04-24"
  comment="Minimal QE-practical UPF v2.0.1 header"
  element="Si"
  pseudo_type="NC"
  relativistic="no"
  is_ultrasoft="F"
  is_paw="F"
  is_coulomb="F"
  has_so="F"
  has_wfc="F"
  has_gipaw="F"
  paw_as_gipaw="F"
  core_correction="F"
  with_metagga_info="F"
  functional="PBE"
  z_valence="4.0"
  total_psenergy="-7.25"
  wfc_cutoff="30.0"
  rho_cutoff="120.0"
  l_max="1"
  l_max_rho="2"
  l_local="0"
  mesh_size="4"
  number_of_wfc="1"
  number_of_proj="1" />
```

### PP_MESH

`PP_MESH` contains `PP_R` and `PP_RAB`. QE reads mesh attributes such as `mesh`,
`dx`, `xmin`, `rmax`, and `zmesh` when present. If `PP_MESH@mesh` differs from
`PP_HEADER@mesh_size`, QE updates the in-memory mesh size from the section
attribute.

Implementation note for this Rust crate: the crate intentionally does not
follow QE's permissive mismatch handling here. `UpfData::validate()` currently
rejects files where `PP_MESH@mesh` and `PP_HEADER@mesh_size` disagree, because
this project prefers stricter structural validation over QE-style recovery.

Example:

```xml
<PP_MESH mesh="4" dx="0.1" xmin="-6.0" rmax="1.2" zmesh="14.0">
  <PP_R>
    0.100000 0.250000 0.600000 1.200000
  </PP_R>
  <PP_RAB>
    0.010000 0.025000 0.060000 0.120000
  </PP_RAB>
</PP_MESH>
```

### PP_NLCC

`PP_NLCC` contains pseudized core charge on the radial mesh. QE reads it only
when `core_correction` is true. Otherwise the in-memory array is allocated and
filled with zeros.

Example:

```xml
<PP_NLCC>
  0.000000 0.000200 0.000500 0.000100
</PP_NLCC>
```

### PP_LOCAL

`PP_LOCAL` contains local potential samples. QE reads it only when `is_coulomb`
is false. The reader tolerates either `PP_NLCC` before `PP_LOCAL` or `PP_LOCAL`
before `PP_NLCC`.

Example:

```xml
<PP_LOCAL>
  -4.000000 -2.500000 -1.000000 -0.250000
</PP_LOCAL>
```

### PP_NONLOCAL

`PP_NONLOCAL` contains projectors, the dense `D_ij` matrix, and augmentation
data for ultrasoft or PAW datasets. If `number_of_proj = 0`, QE returns before
opening this section.

Projectors are numbered `PP_BETA.n` elements. Practical projector metadata
includes `index`, `label`, `angular_momentum`, `cutoff_radius_index`,
`cutoff_radius`, and `ultrasoft_cutoff_radius`. In QE's `pseudo_types.f90`,
projector and wavefunction labels are stored in `CHARACTER(LEN=2)` fields, so
practical labels are short forms such as `3S`, `3P`, `4P`, `3D`. QE also accepts
`norm_conserving_radius` as a compatibility fallback when
`ultrasoft_cutoff_radius` is absent or zero (old UPF v.2 files contained an
error in the tag name). In the v2 read path, QE does not strictly enforce the
`index` attribute against the numbered tag suffix, and a missing
`cutoff_radius` is silently read as `0.0`. This crate preserves the absence as
`None` instead of materializing QE's default `0.0`; when present, a meaningful
`cutoff_radius` should be non-zero.

`PP_DIJ` contains the dense projector coupling matrix.

`PP_AUGMENTATION` is present only for ultrasoft or PAW datasets (when
`is_ultrasoft` or `is_paw` is true). Its metadata is stored as attributes,
including `q_with_l`, `nqf`, `nqlc`, and PAW-specific fields such as `shape`,
`cutoff_r`, `cutoff_r_index`, `augmentation_epsilon`, and `l_max_aug`.

When `q_with_l` is true, augmentation channels use `PP_QIJL.i.j.l` naming.
When `q_with_l` is false, they use `PP_QIJ.i.j` naming. Both carry
`first_index`, `second_index`, and `composite_index` attributes. `PP_QIJL`
entries additionally carry `angular_momentum`.

QE determines channel nullity from the `augmom` or `qqq` values, not from an
`is_null` attribute. Null channels are simply skipped during both reading and
writing. The `is_null` attribute defined in some XSD references is not used by
QE's practical reader or writer.

`PP_Q` is always read when augmentation is present. `PP_MULTIPOLES` is read only
for PAW datasets. `PP_QFCOEF` and `PP_RINNER` are read only when `nqf > 0`
(v2 format only). If `nqlc` is zero, QE defaults it to `2 * l_max + 1`.

Norm-conserving example:

```xml
<PP_NONLOCAL>
  <PP_BETA.1
    index="1"
    label="3S"
    angular_momentum="0"
    cutoff_radius_index="4"
    cutoff_radius="1.2"
    ultrasoft_cutoff_radius="1.2">
    0.120000 0.250000 0.180000 0.040000
  </PP_BETA.1>
  <PP_BETA.2
    index="2"
    label="3P"
    angular_momentum="1"
    cutoff_radius_index="4"
    cutoff_radius="1.2"
    ultrasoft_cutoff_radius="1.2">
    0.000000 0.080000 0.190000 0.070000
  </PP_BETA.2>
  <PP_DIJ>
    -0.500000 0.000000
     0.000000 0.250000
  </PP_DIJ>
</PP_NONLOCAL>
```

Ultrasoft example:

```xml

  pseudo_type="US"
  is_ultrasoft="T"
```

### PP_SEMILOCAL

`PP_SEMILOCAL` is read only when `pseudo_type == "SL"`. Entries are numbered
`PP_VNL.n` elements. QE follows numbering found in existing files rather than
assuming a perfectly mirrored current-writer numbering scheme. Fully
relativistic semilocal reading in this path is effectively unsupported and
returns an error.

Example:

```xml
<PP_SEMILOCAL>
  <PP_VNL.1 l="0">
    -3.000000 -1.800000 -0.700000 -0.150000
  </PP_VNL.1>
  <PP_VNL.2 l="1">
    -2.500000 -1.500000 -0.600000 -0.120000
  </PP_VNL.2>
</PP_SEMILOCAL>
```

### PP_PSWFC

`PP_PSWFC` is opened in the normal read path even when `number_of_wfc = 0`.
Entries are numbered `PP_CHI.n` elements.

QE reads the following attributes from each `PP_CHI.n`:

- `index` (integer): ordinal index, not strictly validated for v2
- `label` (string): conventional short label such as `3S`, `3P`
- `l` (integer): angular momentum
- `occupation` (real): occupation in the reference configuration
- `n` (integer): pseudo-principal quantum number
- `pseudo_energy` (real): single-particle energy in atomic configuration
- `cutoff_radius` (real): inner norm-conserving radius
- `ultrasoft_cutoff_radius` (real): outer ultrasoft radius

When `has_so` is true, QE also reads from `PP_CHI.n`:

- `nn` (integer): principal quantum number (written by QE, read as obsolete)
- `jchi` (real): total angular momentum (l ± 1/2)

QE's writer conditionally includes some attributes:

- `n` is written only when `nchi > lchi` (i.e., the pseudo-principal quantum
  number exceeds the angular momentum).
- `pseudo_energy` is written only when `> 0`.
- `cutoff_radius` is written only when `> 0`.
- `ultrasoft_cutoff_radius` is written only when `> 0`.
- `nn` and `jchi` are written only when `has_so` is true.

Example:

```xml
<PP_PSWFC>
  <PP_CHI.1
    index="1"
    label="3S"
    l="0"
    occupation="2.0"
    n="3"
    pseudo_energy="-0.25"
    cutoff_radius="1.2"
    ultrasoft_cutoff_radius="1.2">
    0.020000 0.110000 0.240000 0.180000
  </PP_CHI.1>
</PP_PSWFC>
```

### PP_FULL_WFC

`PP_FULL_WFC` is read and written only when `has_wfc` is true. It contains
numbered all-electron orbital entries `PP_AEWFC.n`, optional relativistic
all-electron entries `PP_AEWFC_REL.n`, and pseudo entries `PP_PSWFC.n`. QE's
writer sets the section attribute `number_of_wfc` equal to `number_of_proj`.

`PP_AEWFC_REL.n` entries are present only when both `has_so` and `is_paw` are
true. They carry the same attributes as `PP_AEWFC.n` (`index`, `label`, `l`).

QE reads `number_of_proj` entries for each family. The order within the section
is: all `PP_AEWFC.n`, then all `PP_AEWFC_REL.n` (if present), then all
`PP_PSWFC.n`.

Example:

```xml
<PP_FULL_WFC number_of_wfc="1">
  <PP_AEWFC.1 index="1" label="3S" l="0">
    0.030000 0.140000 0.300000 0.220000
  </PP_AEWFC.1>
  <PP_PSWFC.1 index="1" label="3S" l="0">
    0.020000 0.110000 0.240000 0.180000
  </PP_PSWFC.1>
</PP_FULL_WFC>
```

### PP_RHOATOM

`PP_RHOATOM` contains atomic charge density on the current mesh. QE always reads
and writes this section in the normal UPF v2.0.1 path.

Example:

```xml
<PP_RHOATOM>
  0.000100 0.001000 0.005000 0.002000
</PP_RHOATOM>
```

### PP_TAUMOD and PP_TAUATOM

`PP_TAUMOD` and `PP_TAUATOM` are standalone sections read and written only when
`with_metagga_info` is true. They are not wrapped in a container.

Example:

```xml
<PP_TAUMOD>
  0.000000 0.000010 0.000020 0.000005
</PP_TAUMOD>
<PP_TAUATOM>
  0.000000 0.000020 0.000030 0.000006
</PP_TAUATOM>
```

### PP_SPIN_ORB

`PP_SPIN_ORB` is present only when `has_so` is true and the format is v2. It
contains empty numbered metadata entries `PP_RELWFC.n` and `PP_RELBETA.n`. QE
tolerates either wavefunction metadata before projector metadata or the reverse
order.

QE reads `number_of_wfc` `PP_RELWFC.n` entries and `number_of_proj`
`PP_RELBETA.n` entries.

From `PP_RELWFC.n`, QE reads:

- `index` (integer): ordinal index (not strictly validated for v2)
- `nn` (integer): principal quantum number (read but treated as obsolete)
- `jchi` (real): total angular momentum

QE's reader intentionally does not read `lchi`, `els`, or `oc` from
`PP_RELWFC.n`, even though the writer emits them. The comment in
`read_upf_new.f90` explains: "upf%oc is actually missing in some UPF files:
reading it here may spoil the value read earlier and break DFT+U". These values
are already known from `PP_PSWFC`.

QE's writer emits on `PP_RELWFC.n`: `index`, `els`, `nn`, `lchi`, `jchi`, `oc`.

From `PP_RELBETA.n`, QE reads:

- `index` (integer): ordinal index
- `lll` (integer): angular momentum
- `jjj` (real): total angular momentum

QE's writer emits on `PP_RELBETA.n`: `index`, `lll`, `jjj`.

Example:

```xml
<PP_SPIN_ORB>
  <PP_RELWFC.1 index="1" els="3S" nn="3" lchi="0" jchi="0.5" oc="2.0" />
  <PP_RELBETA.1 index="1" lll="0" jjj="0.5" />
</PP_SPIN_ORB>
```

### PP_PAW

`PP_PAW` is present only when `is_paw` is true. Practical section attributes are
`paw_data_format` (integer in QE's `pseudo_types.f90`) and `core_energy` (real).
Practical child sections are `PP_OCCUPATIONS`, `PP_AE_NLCC`, and `PP_AE_VLOC`.

All-electron and pseudo projector-linked wavefunctions are not serialized inside
`PP_PAW` in this QE path; they belong to `PP_FULL_WFC`. Additional PAW working
arrays (`pfunc`, `ptfunc`, `pfunc_rel`) are derived in memory after parsing from
the `PP_FULL_WFC` data and are not part of the wire format.

`PP_OCCUPATIONS` has `number_of_proj` values (full occupation, not only > 0).

Example:

```xml
<PP_PAW paw_data_format="2" core_energy="-10.5">
  <PP_OCCUPATIONS>
    2.000000
  </PP_OCCUPATIONS>
  <PP_AE_NLCC>
    0.000000 0.000300 0.000700 0.000200
  </PP_AE_NLCC>
  <PP_AE_VLOC>
    -5.000000 -3.000000 -1.200000 -0.300000
  </PP_AE_VLOC>
</PP_PAW>
```

### PP_GIPAW

`PP_GIPAW` is present only when `has_gipaw` is true. It carries
`gipaw_data_format` (integer in QE's `pseudo_types.f90`).

The core-orbital subtree uses `PP_GIPAW_CORE_ORBITALS` with
`number_of_core_orbitals` and numbered `PP_GIPAW_CORE_ORBITAL.n` entries. QE
accepts both normal open/close and self-closing core-orbital container forms.
Core orbital attributes include `index`, `label`, `n`, and `l`. In QE's
`pseudo_types.f90`, `gipaw_core_orbital_n` and `gipaw_core_orbital_l` are
`REAL(DP)`, not integers, despite being conceptually quantum numbers.

When `paw_as_gipaw` is true, QE derives GIPAW valence orbitals and local
potentials from PAW, local-potential, and full-wavefunction data in memory. In
this case, only `PP_GIPAW_CORE_ORBITALS` is read from the file;
`PP_GIPAW_ORBITALS` and `PP_GIPAW_VLOCAL` are not present and not read. The
derived arrays are:

- `gipaw_wfs_el` ← `els_beta`
- `gipaw_wfs_ll` ← `lll` (projector angular momenta)
- `gipaw_wfs_ae` ← `aewfc` (from `PP_FULL_WFC`)
- `gipaw_wfs_ps` ← `pswfc` (from `PP_FULL_WFC`)
- `gipaw_vlocal_ae` ← `paw%ae_vloc` (from `PP_PAW`)
- `gipaw_vlocal_ps` ← `vloc` (from `PP_LOCAL`)
- `gipaw_wfs_rcut` ← `rcut` (projector cutoff radii)
- `gipaw_wfs_rcutus` ← `rcutus` (projector ultrasoft cutoff radii)

When `paw_as_gipaw` is false, QE reads `PP_GIPAW_ORBITALS` with numbered
`PP_GIPAW_ORBITAL.n` entries and `PP_GIPAW_VLOCAL` with AE and PS local
potential subsections. The `PP_GIPAW_ORBITALS` container carries
`number_of_valence_orbitals` as an attribute.

GIPAW valence orbital attributes: `index`, `label`, `l` (integer in
`pseudo_types.f90`), `cutoff_radius`, `ultrasoft_cutoff_radius`.

GIPAW valence orbital children: `PP_GIPAW_WFS_AE` and `PP_GIPAW_WFS_PS`.

Example:

```xml
<PP_GIPAW gipaw_data_format="2">
  <PP_GIPAW_CORE_ORBITALS number_of_core_orbitals="1">
    <PP_GIPAW_CORE_ORBITAL.1
      index="1"
      label="2P"
      n="2"
      l="1">
      0.010000 0.050000 0.030000 0.005000
    </PP_GIPAW_CORE_ORBITAL.1>
  </PP_GIPAW_CORE_ORBITALS>
  <PP_GIPAW_ORBITALS number_of_valence_orbitals="1">
    <PP_GIPAW_ORBITAL.1 label="3S" l="0" cutoff_radius="1.2">
      <PP_GIPAW_WFS_AE>
        0.030000 0.140000 0.300000 0.220000
      </PP_GIPAW_WFS_AE>
      <PP_GIPAW_WFS_PS>
        0.020000 0.110000 0.240000 0.180000
      </PP_GIPAW_WFS_PS>
    </PP_GIPAW_ORBITAL.1>
  </PP_GIPAW_ORBITALS>
  <PP_GIPAW_VLOCAL>
    <PP_GIPAW_VLOCAL_AE>
      -5.000000 -3.000000 -1.200000 -0.300000
    </PP_GIPAW_VLOCAL_AE>
    <PP_GIPAW_VLOCAL_PS>
      -4.000000 -2.500000 -1.000000 -0.250000
    </PP_GIPAW_VLOCAL_PS>
  </PP_GIPAW_VLOCAL>
</PP_GIPAW>
```

## Attribute Default Behavior

QE's `get_attr` helper in `xmltools.f90` never errors on a missing attribute.
When an attribute is absent from the XML, the reader silently returns a
type-specific default:

- Integer attributes default to `0`.
- Real attributes default to `0.0`.
- Logical attributes default to `.false.`.
- String attributes default to `''` (empty string).

This crate generally models such absent metadata explicitly when it is
semantically useful. For example, missing `PP_BETA.n/@cutoff_radius` is kept as
`None` instead of being converted to QE's default `0.0`, because a meaningful
cutoff radius is expected to be non-zero when provided.

For real-valued metadata and numeric payload text, this crate also accepts
Fortran double-precision exponent markers `d` / `D` in addition to the usual
`e` / `E` forms. This matches practical QE-oriented files that may contain
values such as `1.d-12` or `0.0125d0`.

This means every attribute in `PP_HEADER` and on numbered entries is effectively
optional from the reader's perspective. However, QE's writer always emits all
header attributes and the standard projector/wavefunction metadata, so
QE-generated files always contain the full attribute set. Third-party generators
may omit attributes that QE would silently default.

## Non-standard Numeric Payload Attributes

The official UPF v2.0.1 prose does not standardize body-formatting attributes
such as `type`, `size`, `rows`, or `columns` on numeric data blocks. However,
the project's bundled fixtures, examples, and git history show that these
attributes often appear in practical QE-oriented files.

These attributes should be treated as compatibility metadata rather than part
of the standard UPF v2.0.1 contract:

- `type`, `size`, and `columns` are commonly emitted on plain numeric blocks.
- `type`, `size`, and `columns` also often appear on structured numeric
  carriers that have real semantic attributes of their own.
- `rows` is mainly seen on matrix-like payloads, especially `PP_DIJ`.

Observed plain numeric blocks:

- `PP_R`, `PP_RAB`, `PP_NLCC`, `PP_LOCAL`, `PP_DIJ`, `PP_Q`,
  `PP_MULTIPOLES`, `PP_QFCOEF`, `PP_RINNER`, `PP_RHOATOM`, `PP_TAUMOD`,
  `PP_TAUATOM`, `PP_OCCUPATIONS`, `PP_AE_NLCC`, `PP_AE_VLOC`,
  `PP_GIPAW_WFS_AE`, `PP_GIPAW_WFS_PS`, `PP_GIPAW_VLOCAL_AE`,
  `PP_GIPAW_VLOCAL_PS`

Observed structured numeric carriers:

- `PP_BETA.n`, `PP_VNL.n`, `PP_CHI.n`, `PP_AEWFC.n`, `PP_AEWFC_REL.n`,
  `PP_PSWFC.n`, `PP_QIJ.i.j`, `PP_QIJL.i.j.l`, `PP_GIPAW_CORE_ORBITAL.n`

In other words, these attributes are common in real files but remain outside
the official UPF v2.0.1 standard. Reverse-engineered QE compatibility should
document them, but schema or model validation should derive actual array sizes
from `PP_HEADER` and section context rather than trusting the attributes.

## Compatibility Rules

- QE is less strict about malformed or missing UPF-v2.0.1 `index` metadata than
  about other structural mismatches.
- QE also tolerates missing real-valued metadata such as `PP_BETA.n/@cutoff_radius`
  by defaulting it to `0.0`.
- This crate accepts Fortran `d` / `D` exponent notation for real values in
  both attributes and whitespace-delimited numeric payloads.
- Logical header attributes are accepted in both compact and XML-like spellings
  observed in real files.
- `PP_INFO` is writer-visible but not part of the structured reader contract.
- QE's v2 reader does not validate cross-references between header flags and
  section presence. If a flag like `has_so` is true but `PP_SPIN_ORB` is
  missing, QE will encounter an XML read error rather than a structured
  validation failure.
